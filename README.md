# dietary_coach_tg_bot
Персональный телеграм-бот для трекинга диеты: дневник питания, замеры тела, PDF-отчёт со статистикой.

# Стек
- Python 3.13 (есть venv в `./venv`)
- aiogram 3.x
- SQLAlchemy + alembic + aiosqlite (SQLite)
- reportlab + matplotlib (PDF-отчёты)
- pillow (парсинг EXIF из caption)
- pytz (часовые пояса)

# Запуск

## Локально
```bash
python -m venv venv
source venv/bin/activate          # Linux/macOS
# .\venv\Scripts\Activate.ps1     # Windows PowerShell
pip install -r requirements.txt
cp .env.example .env              # вписать BOT_TOKEN
python main.py                    # схема БД применится автоматически при старте
```

Переменные грузятся из `.env` (см. `.env.example`). Если `api.telegram.org` недоступен напрямую — задай `PROXY_URL` в `.env` (или экспортируй `HTTPS_PROXY` / `HTTP_PROXY` в shell — оно подхватится).

## В Docker

```bash
cp .env.example .env              # вписать только BOT_TOKEN (DATABASE_URL не нужен)
mkdir -p data exports reports     # SQLite и выгрузки будут жить тут на хосте

docker compose up -d --build      # сборка + старт
docker compose logs -f bot        # логи
docker compose down               # остановить
```

Что монтируется:
- `./data` → `/app/data` — SQLite-файл (`dietary_coach.db`). Переживает пересборку образа.
- `./exports` → `/app/exports` — выгрузки `scripts/export_csv.py`.
- `./reports` → `/app/reports` — PDF-отчёты, если будешь генерить их в контейнере.

Прокси и `BOT_TOKEN` пробрасываются в контейнер автоматически: `HTTP_PROXY` / `HTTPS_PROXY` берутся из shell-окружения хоста (например, из `~/.bashrc`), `BOT_TOKEN` — из `.env`. Миграции прогоняются программно при старте бота (`main.py` сам вызывает `alembic upgrade head`).

Запустить разовую команду в контейнере (например, экспорт CSV):
```bash
docker compose run --rm bot python scripts/export_csv.py --out /app/exports
```

# Проблема
Нужен персональный коуч для ведения дневника питания и сохранения замеров с возможностью вывода статистики. Цель — держать правильное питание, питаться в нужные интервалы времени, не делать перекусов. Нужен трекер, в который удобно отправлять приёмы пищи, голод и насыщение.

# Решение
Телеграм-бот, в который отсылаются фото приёмов пищи, замеры тела, и периодически выгружается отчёт об успехах.

---

# Архитектурные решения
- БД проектируется сразу под мультиюзера: telegram `user_id` присутствует во всех таблицах.
- Запись приёма пищи создаётся **сразу** при отправке сообщения (тип/голод/насыщение = NULL). Нажатие кнопок делает UPDATE.
- Фото не сохраняются на диск — в БД хранится `file_id` и `file_unique_id`.
- Часовой пояс юзера хранится в профиле (спрашивается в `/start`), все таймстемпы — в этом TZ.
- Авторизации нет: любой, кто запустил `/start`, становится юзером.

# Минимальный функционал
- Приём пищи отправляется как фото, набор фото (media group) или текст.
- Сообщение в чат == скинутый приём пищи. Для остального — команды.
- На каждое сообщение приёма пищи появляются кнопки выбора (можно нажать второй раз и поменять).
- Команда `/add_measurements` — последовательный ввод плечи, талия, бёдра, вес.
- Команда `/start` — рост, возраст, пол, вес, обхваты, часовой пояс, цели. Если профиль уже есть — показывает сохранённые данные и блокирует повторное выполнение.
- Команда `/get_report` — PDF-отчёт за период `DDMMYYYY-DDMMYYYY`.
- Бот не сохраняет фото на диск, только идентификаторы.
- Время приёма пищи: парсим EXIF из caption/подписи если задано, иначе `message.date` в TZ профиля. Также есть inline-кнопка "Изменить время" с календарём.
- Можно отправить забытый приём пищи: задать дату/время через inline-календарь (или указать в caption).
- Нет авторизации: любой `/start` создаёт юзера.
- Скрипт `scripts/export_csv.py` для конверта SQLite → набор CSV.

# Будущий функционал
- Полноценный мультиюзер (UI/команды), сейчас только схема готова.
- Напоминания о замерах тела (раз в 2 недели).
- OpenRouter / Gemini для анализа фото и текстовых советов.
- `/chat` — диалог с LLM с контекстом БД.
- Экстракт ингредиентов / гарвардская тарелка / КБЖУ из фото.
- Выбор личности / метода общения LLM.

# UI

## /start
Последовательный FSM-диалог:
1. рост (см)
2. возраст
3. пол (м/ж)
4. вес (кг)
5. обхваты: плечи, талия, бёдра (см)
6. часовой пояс (IANA, например `Europe/Moscow`)
7. цели (inline-кнопки, можно несколько)

В конце — агрегированное сообщение с выбором `[Сохранить] [Отмена]`.

После сохранения — короткая памятка:
- что такое дневник питания
- шкалы голода и насыщения
- советы по гарвардской тарелке
- как делать замеры
- как пользоваться ботом (команды)

Если профиль уже существует, `/start` показывает сохранённые данные и не запускает диалог повторно.

### Список целей
Взаимоисключающие (можно выбрать только одну):
- Снижение веса
- Набор веса
- Удержание веса

Независимые (любое количество):
- Уменьшение талии
- Увеличение плеч
- Увеличение бёдер
- Регулярное питание (одно время каждый день)
- Меньше перекусов
- Не пропускать основные приёмы пищи
- Не переедать (избегать насыщения 4-5)
- Не быть сильно голодным (избегать голода 4-5)
- Регулярный спортпит

При попытке выбрать конфликтующую цель — предыдущая снимается, юзер получает уведомление.

## Отправка приёма пищи
Скинули текст / картинку / набор картинок (media group) → у сообщения появляются 4 ряда кнопок (один выбор в ряду):
- `[завтрак] [обед] [ужин] [перекус] [спортпит]`
- голод: `[1] [2] [3] [4] [5]`
- насыщение: `[1] [2] [3] [4] [5]`
- `[Изменить время]` → inline-календарь для смены времени приёма пищи

Время приёма определяется так:
1. Если в caption указано `@DD.MM.YYYY HH:MM` — парсится оттуда (для забытых).
2. Иначе — `message.date`, конвертированное в TZ профиля.
3. Юзер может переписать кнопкой "Изменить время".

## Шкалы голода и насыщения
- Голод: `1 = сыт`, `5 = очень голоден`.
- Насыщение: `1 = не наелся`, `5 = переел`.

## /add_measurements
Последовательно: плечи → талия → бёдра → вес. В конце агрегированное сообщение и кнопки `[Сохранить] [Отмена]`.

## /get_report
Юзер вводит период `DDMMYYYY-DDMMYYYY`. Бот возвращает PDF-отчёт.

Что в отчёте:
- сводная таблица: изменения веса и замеров (с цветами зелёный/красный по целям; изменения меньше погрешности не подсвечиваются).
- графики: динамика веса и замеров, разброс времени приёмов пищи, количество перекусов в день, распределение голода/насыщения, регулярность спортпита.

Цели, поддерживаемые в отчёте:
- разброс времени основных приёмов пищи (завтрак, обед, ужин)
- пропуски основных приёмов пищи
- количество перекусов
- регулярность спортпита
- переедание (насыщение 4-5)
- голод (4-5)
- успех насыщения (насыщение > 1)
- динамика веса и замеров (под цели снижения/набора/удержания)

Не поддерживается: сахар, гарвардская тарелка (будущий функционал).

# UX
Покушал → скинул фото → жамкнул кнопки.
Раз в две недели → `/add_measurements`.
Периодически → `/get_report`.
